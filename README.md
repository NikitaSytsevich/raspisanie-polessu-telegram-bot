# Расписание ПолесГУ — Telegram-бот

Самостоятельный Telegram-бот для расписания спортивных объектов ПолесГУ. Это отдельный Vercel-проект: он не обращается к веб-приложению, не использует его API, Blob, cron или пользовательские смены.

## Что делает

- `/today`, `/tomorrow`, `/week` — расписание с навигацией кнопками;
- загружает данные напрямую с официальных страниц ПолесГУ при запросе;
- отправляет Rich Messages Bot API 10.1: заголовки, раскрываемые дни недели и таблицы «время / что проходит»;
- не хранит chat ID и не ведёт рассылок: никаких подписок, настроек, cron-задач и состояния.

Такой состав бережно расходует лимиты Vercel Hobby: нет фоновых запусков, базы данных и регулярной рассылки, а четыре страницы источника загружаются параллельно только по команде пользователя. В памяти тёплого инстанса есть короткий 90-секундный кэш, который не является хранилищем и не влияет на работу после cold start.

## Деплой

1. Создайте **отдельный** GitHub-репозиторий из этой папки и импортируйте его в Vercel как новый проект.
2. В Vercel → Settings → Environment Variables добавьте значения из `.env.example`:
   - `TELEGRAM_BOT_TOKEN` — токен от @BotFather;
   - `TELEGRAM_WEBHOOK_SECRET` — новая случайная строка из букв, цифр, `_` и `-`;
   - `APP_URL` — URL именно нового Vercel-проекта.
3. Сделайте redeploy. Однократно откройте в браузере:

   `https://<ваш-домен>/api/telegram?setup=<TELEGRAM_WEBHOOK_SECRET>`

   В ответе должно быть `"ok": true`. Эта операция зарегистрирует новый webhook и команды бота; старый webhook будет автоматически заменён.
4. Напишите боту `/start`.

Не передавайте ссылку из шага 3 другим людям и не сохраняйте её в истории публичных каналов: секрет в ней нужен только для начальной настройки. После успешной регистрации можно заменить его новой переменной и повторить setup.

## Локальная проверка

```bash
npm install
npm test
npx vercel dev
```

Публичный HTTPS URL нужен Telegram для webhook, поэтому локальный сервер сам по себе бот не подключит.

## Почему нет уведомлений

Регулярный парсинг четырёх внешних страниц и рассылка требуют постоянного хранилища подписок и расходуют лимиты вызовов и вычислений. Это отдельная задача, для которой нужна инфраструктура с очередью, наблюдаемостью и бюджетом на фоновые процессы. Здесь оставлено надёжное ядро — быстрый запрос актуального расписания по инициативе пользователя.
